Довантаження **повної історії** 5 найбільших процесів (замість bounded-семплу по
500 з батча 012). Ключ — партиціювання за префіксом `Number` через `startswith`,

## Метод — партиціювання за Number

- **Формат `Number`**: `ДО-` + 42-знакове поле, послідовність вирівняна праворуч
  (напр. `ДО-000…000178`), довжина 45. `ДО-` у percent-encoding = `%D0%94%D0%9E-`.
- **Чанк k** = `startswith(Number, 'ДО-' || lpad(k,39,'0')) eq true` → рядки з
  seq у `[k*1000 .. k*1000+999]` (≤1000) **одним запитом** (ТЧ вкладені у фід).
  `k` = 0..~186 для Согласования.
- **Стоп** — 5 порожніх чанків поспіль (gap-safe: історія має проміжки від
  видалених/непозначених; перший чанк Утверждения теж порожній — seq стартує з ~1047).
- **Резюмовність** — `RSD_LOAD_LOG(entity, chunk_k, rows_got)`: повторний запуск
  переграє залоговані чанки без повторного fetch.

## Артефакти

- `loader-pkg.sql` — пакет `RSD_BP_ARCHIVE.load(entity, enc, count)` + control-таблиця
  `RSD_LOAD_LOG`. Fetch із 3 ретраями (`dbms_session.sleep`), MERGE у `RSD_ODATA_RAW`.
- `load-all.sql` — драйвер: 4 сутності (Утверждение завантажено окремо в тесті),
  кожна у блоці з fresh `apex_session`.
- `verify.sql` — header count == `$count`, ТЧ-масштаб, 0 INVALID.
- Проекція RAW→типізовані+ТЧ — **reused `migration/012-bp-data-bounded/project.sql`**
  (усі 5 сутностей + 30 ТЧ; ідемпотентна delete+insert по OWNER_ID).

## Виконання (пастки)

- **Detached-запуск обовʼязковий**: повний прогін (RAW ~хв, проекція ~десятки хв)
  перевищує foreground-таймаут. `docker exec -d ... sqlplus > /tmp/*.log`; моніторинг
  через `RSD_LOAD_LOG` (real-time, per-chunk commit) + лог-маркери. Foreground
  sqlplus, вбитий по таймауту, НЕ комітить буфер → «нічого не завантажено».
- **`DBMS_LOCK` не виданий BAS_REVERSE** → sleep через `DBMS_SESSION.SLEEP` (26ai, public).
- **RAW-обсяг**: 365k записів з ТЧ ≈ ~1.5–2 ГБ CLOB у `RSD_ODATA_RAW`; USERS
  autoextensible, Free-ліміт 12 ГБ, запас достатній. RAW лишено (re-projectable).
- Payload чанку (1000 записів з усіма ТЧ) ~кілька МБ; `transfer_timeout=600`.
- **12 ГБ Free-ліміт (ORA-12954)**: RAW.doc LOB роздувся до **~9.3 ГБ** (365k записів
  з вкладеними ТЧ) → проекція Ознакомление/Исполнение впала на ліміті. RAW — staging,
  тож після проекції великих: `delete` спроектованих + reclaim. **`SHRINK SPACE` НЕ
  спрацював** (LOB retention тримає видалене місце) → **`TRUNCATE ... DROP ALL STORAGE`**
  (миттєвий reclaim). Підсумок: **10741 → 1368 МБ**. Порядок проекції: спершу великі
  (Согласование, КомплексныйПроцесс), reclaim, потім решта у вивільнений запас.
- **§9 поліморфні ТЧ-поля не відновлені для повного архіву.** Проекція
  (`section_block`) лишає `*_REF_TYPE/_REF_ID` (Исполнитель, Предмет…) **NULL** — це
  §9-стан за замовчуванням. Резолв Роль/Користувач у стор. 12 app-201 був **окремим
  UPDATE з RAW** (не в `project.sql`); повна ре-проекція скинула його на 500-семплі →
  у app 201 «Тип/Виконавець» тепер порожні для всього архіву (213 675 рядків
  ISPOLNITELI, ref_type/ref_id = 0). RAW truncated → відновлення потребує re-fetch
  поля `Исполнитель`+`_Type` для Согласования + resolve-UPDATE. Кандидат у §9-бэклог
  (як 4b): резолв поліморфних ТЧ повного архіву.
