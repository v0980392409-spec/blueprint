# Батч 004 — живі дані НСІ з OData

Трек **даних** (окремий від структури): наповнення готових таблиць НСІ живими
записами з OData BAS. Пілот — Організації → `RSD_ORGANIZATIONS`.

## Механізм

- **Джерело**: `https://rv.entercom.ua:5683/bas_doc/odata/standard.odata/` (Документообіг).
  Опубліковано 251 `Catalog_*`; усі 6 ключових НСІ доступні (Организации,
  Контрагенты, Пользователи, ФизическиеЛица, БанковскиеСчета, Валюты).
- **Транспорт**: `apex_web_service.make_rest_request` з Web Credential
  `BAS_DOC_CRED` (створений у першому експерименті; ACL до хоста вже є —
  на ньому працював `bas_sync_pkg`). Виклику потрібен контекст APEX-сесії
  (`apex_session.create_session`), інакше credential недоступний.
- **Парсинг**: `JSON_TABLE` по `$.value[*]`; кириличні поля — у лапках у шляху
  (`'$."ИНН"'`).
- **Реконсиляція**: `MERGE ... ON (t.LEGACY_REF = s.Ref_Key)` — ідемпотентно;
  повторний запуск оновлює, не дублює. `Ref_Key` (UUID 1С) — той самий ключ,
  що лежить у `LEGACY_REF` структури.
- **Enum-маппінг**: рядкове значення `ЮрФизЛицо` («ЮрЛицо») → `ENTITY_KIND_ID`
  підзапитом до `RSD_ENUMS` (`enum_type='ЮрФизЛицо' and value_key=<val>`) —
  замикає ланцюг батчів 002 (перечислення) + 001 (структура) на живих даних.

## Результат пілоту (`load-organizations.sql`)

- **158 організацій** завантажено (MERGE), усі 158 з коректним `ENTITY_KIND_ID`
  → «Юридична особа» через FK на `RSD_ENUMS`. Кирилиця ціла.
- FK на Контрагентів/Банківські рахунки (`DEVELOPER_ID`, `MAIN_BANK_ACCOUNT_ID`)
  у джерелі — UUID; поки NULL. Замкнуться після завантаження тих довідників +
  прохід реконсиляції UUID→ID (наступний крок треку даних).

## Пастки (усунуті)

- **CODE/ENTITY_KIND_ID → NULLABLE.** Реальні дані мають порожній `Code` і не
  завжди заповнений вид → NOT NULL валив завантаження. Це підтверджує принцип
  батча 003: обмеження 1С (унікальність коду, FillChecking) — рівня UI, не БД;
  для історичних даних вони мʼякі, enforce в APEX. `ddl.sql` батча 001 оновлено.
- **`''` = NULL в Oracle** — порожні OData-рядки через `nullif(x,'')`.

## Ядровий набір (генератор)

Решта 5 ядрових довідників — детермінованим генератором
`bas2apex/tools/gen_odata_loaders.py` за `field-map.json` (карта OData-поле →
колонка, яку emit-ить `gen_catalog_batch`). Двофазно:

- `raw-ddl.sql` — staging `RSD_ODATA_RAW(entity, ref_key, doc)` (сирий JSON).
- `stage.sql` — сторінкова вибірка кожного `Catalog_*` у RAW ($top=500 —
  більше валить `apex_web_service` на великих записах, див. пастку).
- `project.sql` — типізований MERGE RAW → `RSD_*`; **запускати двічі**
  (реконсиляція циклів: посилання резолвляться проти вже завантажених таблиць).
- `reconcile-organizations-refs.sql` — доставляє Організації в RAW і замикає
  їхні `DEVELOPER_ID`/`MAIN_BANK_ACCOUNT_ID` після завантаження цілей.

Завантажено: Пользователи 662, ФизическиеЛица 800, Валюты 39, Банківські
рахунки 5752, Організації 158 — **повністю**; Контрагенти 3480 з 10597 —
**частково** (пагінація великих сутностей 1С OData, див. пастку).

## Пастки (усунуті)

- **CODE/ENTITY_KIND_ID/OWNER_ID → NULLABLE.** Реальні дані мають порожній
  Code, не завжди заповнений вид/власника → NOT NULL валив завантаження.
  Обмеження 1С (унікальність, FillChecking, підпорядкування) — рівня UI,
  enforce в APEX. `''` = NULL в Oracle → `nullif`.
- **Глибокий `$skip` не працює для великих сутностей.** `apex_web_service`
  через `make_rest_request` обривався на ~3000 рядках (skip≈3000): у 1С OData
  глибокий `$skip` дорогий і запит не вкладався в таймаут → короткий/обрізаний
  відповідь → цикл виходив рано (3019 з 10597 Контрагентів). curl напряму
  віддавав усі сторінки — проблема саме в парі deep-skip × timeout.
  Keyset-пагінація (`$orderby=Ref_Key asc` + `$filter=Ref_Key gt '<курсор>'`,
  фільтр — голий uuid у лапках, без `guid`) знімає deep-skip, але впирається в
  **інше**: 1С упорядковує GUID у БІНАРНОМУ порядку, а курсор `max(Ref_Key)`
  порівнюється лексикографічно (рядок), і `$orderby=Ref_Key asc` 1С фактично
  не застосовує — курсор розсинхронізується → рання зупинка (3480 з 10597).
  Надійне рішення для великих сутностей — пагінація за природним НЕ-GUID
  ключем (`Code`/номер) або серверна вивантаження (ORDS-side / `bas_sync_pkg`).
  Малі й середні сутності (≤~5–6k, як тут 5 із 6) вантажаться повністю.
- **Поліморфний власник (`Owner` + `Owner_Type`).** Банківські рахунки
  належать різним типам (Контрагенти/Організації). Один FK `OWNER_ID` це не
  виражає → лишаємо NULL, у §9 (як композитний тип). `OWNER_ID` зроблено
  nullable у генераторі.
- **SSH-сесія рветься на довгому MERGE** (exit 255) → відкат. Довгі
  завантаження запускати detached на VPS (`nohup`), опитувати лічильники.
