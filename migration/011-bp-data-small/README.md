# Батч 011 — дані ТЧ процесів з OData (капабіліті + демо малих)

Демонстрація завантаження **header + табличних частин (ТЧ)** документів/процесів
з OData — вкладені масиви `$.<Section>[*]` → дочірні таблиці. Документи ДО в
цьому OData порожні (7×404, 1 порожній); дані є лише в BusinessProcess.

## Що завантажено (= OData $count, 0 ORA)

3 малих процеси **повністю (header + усі ТЧ)**:

| Процес | Header | ТЧ (приклади рядків) |
|---|--:|---|
| РешениеВопросовВыполненияЗадач | 19 | PREDMETY 19, TSIKLYRASSMOTRENIYA 8 |
| Рассмотрение | 4 | ISPOLNITELIISPOLNENIYA 2, … |
| ОбработкаВнутреннегоДокумента | 2 | PREDMETY, … |

ТЧ-рядки: `OWNER_ID`→батько, `LINE_NO` (ordinality), скаляри/enum/ref резолвлені
(напр. header `SOSTOYANIE_ID` 19/19). Поліморфні ТЧ-поля (Предмет/Исполнитель) —
NULL (§9, як у головних таблицях).

## Що додано в конвеєр (переиспользовно)

- **`gen_document_batch`**: `tab_section` реєструє field-map ТЧ під
  `parent["sections"]` (odata-ім'я секції, дочірня таблиця, поля).
- **`gen_odata_loaders`**:
  - `proj_expr(f, src=...)` — параметризовано джерелом значення (для ТЧ — колонка
    JSON_TABLE `jt.cN` замість `json_value(doc,...)`).
  - `section_block` — проекція ТЧ: `JSON_TABLE(doc, '$."<Section>"[*]' columns(...))`
    → `insert <child>(OWNER_ID, LINE_NO, …)` (ідемпотентно: delete+insert).
  - **`enc(entity_ref)`** — префікс OData entity set з типу (`Document_`/
    `BusinessProcess_`, не лише `Catalog_`). Був баг: усі не-довідники йшли як
    `Catalog_<Name>` → 404.
