# Батч 012 — bounded-вибірка 5 великих процесів (по 500 + ТЧ)

Семпл історичних даних 5 найбільших BusinessProcess (замість повного архіву на
185k+). Демонструє bounded-завантаження header+ТЧ великих сутностей.

## Що завантажено (0 ORA)

| Процес | Header | OData $count (усього) |
|---|--:|--:|
| Согласование | 500 | 185 044 |
| КомплексныйПроцесс | 500 | 108 376 |
| Ознакомление | 500 | 50 427 |
| Исполнение | 500 | 11 942 |
| Утверждение | 500 | 9 508 |

**2500 header** + їхні ТЧ (напр. SOGLASOVANIE_ISPOLNITELI 843,
_REZULTATYSOGLASOVANIYA 702, OZNAKOMLENIE_ISPOLNITELI 508). ТЧ: OWNER_ID→батько,
LINE_NO, скаляри/enum/ref резолвлені; поліморфні поля NULL (§9).

## Нове в конвеєрі

- **`--limit N`** у `gen_odata_loaders`: bounded-вибірка одним `$top=N` запитом
  (без пагінації) — для семплу великих сутностей. Обрано 500 (payload ~2.6MB, ОК).
- Fix репортингу bounded: `dbms_output` до `commit` (після commit `sql%rowcount`=0 —
  дані вантажились, але лог брехав «bounded 0»).

## Повний архів (за потреби)

>3000 → deep-$skip таймаутить; для повного завантаження великих процесів —
Code-prefix пагінація за `Number` (як stage-kontragenty-by-code.sql), header ×
4–7 ТЧ = мільйони рядків, години. Цей батч — свідомий семпл 500.
