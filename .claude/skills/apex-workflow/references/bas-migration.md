# Миграция BAS-маршрутов (BusinessProcess) → APEX Workflow

Как переносить «маршруты согласования» BAS (1С Документооборот) в APEX Workflow 26.1 — или
строить заново. Опирается на записку [`docs/businessprocess-workflow.md`](../../../../docs/businessprocess-workflow.md)
и проверенный на стенде пилот (батчи 009/013, `migration/013-workflow-soglasovanie-build/`).
Модель движка — в [concepts.md](concepts.md); API и рецепт сборки — в
[build-and-code-first.md](build-and-code-first.md).

---

## 1. Главное отличие маршрута от справочника/документа

Справочник/документ/регистр мигрируются **механически** конвейером `bas2apex` (атрибуты →
таблицы). Маршрут — **нет**: сам маршрут (точки, ветвления, условия, эскалации) **не лежит в
атрибутном XML-дампе** (`props=[]`). Он живёт в графической схеме БП 1С и в BSL-обработчиках
(17–56 штук на процесс). Поэтому маршрут — **проектная работа**, а не автоген.

### Двухслойная модель

- **Слой 1 — состояние/данные (механически, как документ).** Экземпляры BusinessProcess →
  таблица-заголовок + ТЧ (`gen_document_batch`); `Task` ЗадачаИсполнителя → таблица рабочих
  заданий; enum-состояния (`СостоянияБизнесПроцессов`, `РезультатыСогласования`…) → `RSD_ENUMS`.
  Это **уже сделано** для всех 13 процессов (батчи 009/010).
- **Слой 2 — маршрут/логика (проектирование).** Реконструировать маршрут по структуре + enum-
  результатам + именам BSL-обработчиков, собрать APEX Workflow. Это и есть предмет этого файла.

## 2. Таблица соответствий BAS → APEX Workflow

| BAS (1С) | APEX Workflow 26.1 |
|---|---|
| `BusinessProcess` (вид процесса) | **Workflow Definition** (+ версия) |
| Экземпляр процесса | **Workflow Instance** (стартует на строке таблицы процесса) |
| Точка маршрута (действие) | **Activity** (`Execute Code` / `Human Task - Create` / `Workflow Switch`) |
| Точка маршрута (условие/разветвление) | **`Workflow Switch`** + **Branches** (по переменной) |
| Обработка ошибки перехода | **Error Connection** (по SQL-коду) |
| `ЗадачаИсполнителя` (Task) | **Human Task** activity + **Task Definition** (Approval/Action) |
| Исполнитель = `Роль` \| `Пользователь` | **Participants / Potential Owners** задачи (Value Type = SQL/Static) |
| `РезультатСогласования` (Погоджено/Ні/…) | исход задачи → авто-переменная `TASK_OUTCOME` (+ `APPROVER`) |
| `СостоянияБизнесПроцессов` (Активний/…) | **Workflow state** (`Active`/`Suspended`/…) + прикладной `SOSTOYANIE_ID` |
| `Предмет` (полиморф, 12 типов) | Details Primary Key + Workflow **Variable** (`REF_TYPE`/`REF_ID`) + deep-link |
| BSL-обработчик перехода | **`Execute Code`** activity (PL/SQL) **или** Task Definition Action |
| Эскалация / сроки (`Срок`) | `Due On` активности + **Timeout Connection** + Expiration Policy задачи |
| `ВариантСогласования`: По очереди / Всем сразу / Смешанно | серийно = цепочка Human Task; параллельно = **Parallel Flow** |

> Внутренние русские имена 1С (`СостоянияБизнесПроцессов`) — технический ключ, не переводим.
> Подписи UI — дословно из BAS (украинский). Идентификаторы БД — латиница, префикс `RSD_`.

## 3. Канонический шаблон маршрута (из пилота «Согласование»)

Линейный approve/reject маршрут собирается так (проверено на app 200, процесс ID=94):

```
Start
  → [Execute Code] «Позначити активним»   -- SOSTOYANIE_ID = 768 (Активний)
  → [Human Task - Create] «Погодження»     -- Task Definition Approval; исход → TASK_OUTCOME
  → [Workflow Switch] по TASK_OUTCOME       -- ветвление APPROVED / REJECTED
        ├─ branch APPROVED → [Execute Code] «Погоджено»  -- REZULTAT = 747 + строка ТЧ
        └─ branch REJECTED → [Execute Code] «Не погоджено» -- REZULTAT = 746 + строка ТЧ
  → End
```

> **Важная поправка к рецепту батча 013.** Батч 013 (по незнанию нативного паттерна) делал
> маршрут *линейным* и выбирал 747/746 *в коде* последней активности по своей переменной
> `V_OUTCOME`. Это работает, но **не канонично**. Правильно: approval-активность **сама**
> заполняет переменные `TASK_OUTCOME` (`APPROVED`/`REJECTED`) и `APPROVER`, а ветвление делает
> активность **`Workflow Switch`** (Check Workflow Variable = `TASK_OUTCOME`). Так ветки видны
> на диаграмме и соответствуют «точкам маршрута» 1С. Линейный вариант держать как fallback,
> если Switch по каким-то причинам недоступен.

Привязка версии: **Additional Data → Source = Table `RSD_SOGLASOVANIE`, PK = `ID`** — тогда в
активностях доступны bind строки процесса: `:APEX$WORKFLOW_DETAIL_PK` (= `:ID`), и колонки
через `Additional Workflow Data` (SQL-запрос по `:APEX$WORKFLOW_DETAIL_PK`).

## 4. Процедура миграции одного процесса

1. **Слой 1 (данные)** — убедиться, что таблицы процесса и `Task` уже на стенде (батчи
   009/010) и enum-состояния/результаты — в `RSD_ENUMS` (батч 002).
2. **Реконструкция маршрута** — из трёх источников (графической карты в дампе нет):
   структура ТЧ (`Исполнители`, `Результаты*`), значения enum-результатов, имена BSL-
   обработчиков. Зафиксировать точки/ветки/условия в README батча (+ mermaid).
3. **Task Definition** — тип `Approval Task` (2 исхода Approved/Rejected) для «согласования/
   утверждения»; `Action Task` — для «ознакомления/исполнения». Subject `&NAIMENOVANIE.`,
   Static ID **латиницей** (пастка). Участники — см. п.5.
4. **Workflow** — собрать по шаблону п.3 (Designer или код-first — [build-and-code-first.md](build-and-code-first.md)).
   Static ID **латиницей**. Источник = таблица процесса, PK = `ID`.
5. **Участники** — пилот: статические `CLAUDE`/`VIKTOR` (заглушка). Продакшн: **identity-bridge**
   BAS→APEX — сопоставить `RSD_POLZOVATELI.IDENTIFIKATORPOLZOVATELYAIB` с логинами APEX
   (ещё не сделано; `Catalog.ПолныеРоли` вообще не мигрированы). Potential Owner через
   `Value Type = SQL Query` из `RSD_<PROC>_ISPOLNITELI`.
6. **Активировать** версию (без участников Active не активируется) → **headless-тест**
   (`sql/pilot-test.sql`: `start_workflow` → `complete_task APPROVED` → ждём `REZULTAT=747`).
7. **Зафиксировать в репо** — workflow **не** держится APEXLang'ом → отдельный SQL-трек:
   `apex export -applicationid <app> -expType APPLICATION_SOURCE`, положить `fNNN.sql` в
   `migration/NNN-.../export/`.
8. **Тиражировать** шаблон на родственные процессы (те же ТЧ Исполнители+Результаты):
   **Утверждение / Рассмотрение / Ознакомление / Регистрация**.

## 5. Что механически, что проектно

| Механически (конвейер / шаблон) | Проектно (руками, per-process) |
|---|---|
| Таблицы процесса + ТЧ, `Task`, enum-состояния | Реконструкция маршрута (нет графкарты) |
| Канонический approve/reject шаблон (п.3) | Классификация 17–56 BSL-обработчиков |
| `Execute Code` для смены `SOSTOYANIE_ID`/`REZULTAT` | Динамические этапы (`КомплексныйПроцесс`, 56 обр.) |
| Enum-значения результатов (747/746/…) | Мультиинстанс по `ВариантСогласования` → Parallel Flow |

BSL классифицировать: назначение исполнителей → participants SQL; проверки заполнения →
Task validation / Execute Code; автопереходы → Switch/Execute Code; доменная логика → руками.

## 6. Известные пробелы и решения

- **Нет графических маршрутных карт** 13 БП в дампе → маршрут реверсим по структуре+enum+BSL.
  Идеально — отдельный экспорт схем БП из 1С (запросить у владельца).
- **Нет identity-bridge** BAS→APEX → участники пока статические. Продакшн требует сопоставления
  пользователей/ролей 1С с аккаунтами APEX (+ регистр ВЕРХНИЙ, иначе `ORA-20987`).
- **Третий исход** `Погоджено із зауваженнями` (748) недоступен в Approval Task (только 2) →
  кастомный маршрут или Action Task с доп. логикой.
- **Мультиинстанс** (По очереди 188 / Всем сразу 189 / Смешанно 190; итерации
  `ПовторитьСогласование`) → серия Human Task или Parallel Flow — следующий виток.
- **workflow не в APEXLang** (`applications/*` его не держит) → фиксировать SQL-экспортом;
  реимпорт `.apx` его **сотрёт**. Код-first авторинг возможен (см. build-and-code-first.md).

## 7. Приоритет процессов (рекомендация)

Базовый документооборот сначала: **Согласование** (пилот, готов) → **Утверждение** →
**Рассмотрение** → **Ознакомление** → **Регистрация**. Сложные (`КомплексныйПроцесс` —
динамические этапы, 56 обработчиков; `Приглашение`; `Поручение`) — последними, возможно
частично. Один процесс = один прогон = один батч `migration/NNN-...`.
