# §9 — чек-лист архітектурних рішень (батчі 003 + 005)

Зведення **173 пунктів §9** (батч 003 — 109, батч 005 — 64) у **8 груп**. Кожна
група = одне рішення, а не N однакових. Заповни колонку **Рішення** — під неї
згенерую зміни DDL/маппінгу (де потрібно) і оновлю глосарій/генератори.

Легенда впливу: 🟢 підтвердити дефолт (0 змін) · 🟡 зміна схеми (DDL + подекуди
backfill даних) · ⚪ бэклог логіки (не схема).

---

## 1. 🟢 Композитні типи → `REF_TYPE`/`REF_ID` — **40 реквізитів / 34 таблиці**

Реквізит із ≥2 типами посилань (напр. `ВариантыОтчетов.Отчет`,
`...НастройкаУведомлений.Получатель`, `КлючТипа`). Зараз: поліморфна пара
`<C>_REF_TYPE VARCHAR2(60)` + `<C>_REF_ID NUMBER`, **без DB-FK**. Значення
резолвиться у застосунку за (тип, id).

- **Опція A (дефолт):** лишити `REF_TYPE`/`REF_ID` скрізь. Гнучко, однаково, вже реалізовано.
- Опція B: для наборів ≤3 типів — окремі nullable FK-колонки (`<C>_КОНТРАГЕНТ_ID`, `<C>_ОРГАНИЗАЦИЯ_ID`…). DB-цілісність, але ширша схема й розгалуження в UI.

**Рекомендація:** A (глобально). B — точково лише там, де критична FK-цілісність.
**Рішення:** ✅ **A — REF_TYPE/REF_ID скрізь** (власник, 2026-07-19). 0 змін.

## 2. 🟡 Поліморфний власник `OWNER_ID` — **2 таблиці**

`OWNER_ID` зараз одна NUMBER-колонка, але власник двотиповий → тип втрачено:
- `RSD_BANKOVSKIESCHETA` — Контрагенти **|** Організації · ⚠️ **вже завантажено 5752 рядки** (потрібен backfill `OWNER_TYPE` з OData `Owner_Type`).
- `RSD_NASTROI_KIRABOCHIKHMESTPOLZOVATELEI` — Користувачі **|** РоліВиконавців.

- **Опція A (дефолт, рек.):** додати `OWNER_TYPE VARCHAR2(60)` поряд з `OWNER_ID` (як композит). Для рахунків — backfill з живих даних.
- Опція B: дві окремі nullable FK (`KONTRAGENT_ID`/`ORGANIZATSIYA_ID`).

**Рекомендація:** A — мінімальна зміна, знімає двозначність; backfill рахунків тривіальний.
**Рішення:** ✅ **A — +OWNER_TYPE + backfill** (власник, 2026-07-19). → батч 007.

## 3. 🟢 Поліморфні виміри регістрів (у ключі) — **28 вимірів**

Вимір регістра без типу — `Объект`/`Владелец`/`БизнесПроцесс`/`Ссылка`/`Предмет`
(напр. `ВерсииОбъектов.Объект`, `НастройкиУведомлений.Объект`). Зараз:
`<C>_REF_TYPE`/`<C>_REF_ID` **у складі UNIQUE-ключа** (вимір не можна прибрати).

- **Опція A (дефолт):** лишити полі-ключ REF_TYPE/REF_ID. Faithful, ідентичність рядка збережена.
- Опція B: сурогатний ключ лише за `_REF_ID` (ігнорувати тип) — ризик колізій між типами.

**Рекомендація:** A.
**Рішення:** ✅ **A — REF_TYPE/REF_ID у ключі** (власник, 2026-07-19). 0 змін.

## 4. 🟡 Значення додаткових реквізитів (характеристики) — **33 колонки**

`Значение` у ТЧ `*_DOPOLNITELNYEREKVIZITY` / `*_ZNACHENIYADOSTUPA` — нетипізоване
(тип задає властивість, патерн 1С `ДополнительныеРеквизитыИСведения` =
ChartOfCharacteristicTypes). Зараз колонку **не створено** (drop, §9).

- **Опція A (рек.):** типізовані сестри — `VALUE_STRING VARCHAR2`, `VALUE_NUMBER NUMBER`, `VALUE_DATE DATE`, `VALUE_BOOL`, `VALUE_REF_TYPE`/`VALUE_REF_ID` + `VALUE_TYPE`. Faithful, APEX показує/редагує за типом властивості.
- Опція B: одна `VALUE_TEXT VARCHAR2(4000)` + `VALUE_TYPE` (стрінгіфікація; втрата типів/посилань).
- Опція C: відкласти (не заповнювати доп.реквізити на цьому етапі).

**Рекомендація:** A для ТЧ `*_DOPOLNITELNYEREKVIZITY` (справжні доп.реквізити);
для `ЗначенияДоступа`/`Назначение` — окремо (там значення = посилання доступу).
**Рішення:** ✅ **A — типізовані VALUE_*** (власник, 2026-07-19). → батч 007.

## 7. ⚪ BSL-обробники — **63 обʼєкти / 42 імені** (не схема, бэклог логіки)

Розподіл за призначенням (реімплементація в APEX):
- `ПередЗаписью`(50)/`ПриЗаписи`(35) — дефолти/денормалізація/валідація → APEX-процеси або DB-тригери.
- `ОбработкаЗаполнения`(21)/`...ПроверкиЗаполнения`(16)/`НеЗаполненоПоле`(2)/`Проверить...Предопределенного`(3) → **APEX-валідації**.
- `ПриКопировании`(12) → логіка дії «копіювати».
- `ПередУдалением`(4)/`ПометкаУдаленияВИБ`(2) → мʼяке видалення вже є; реф-перевірки → APEX.
- Доменні одиничні (`НайтиШаблонЗадачЧекЛиста`, `ПроиндексироватьНастройки`…) → ручний розбір при дизайні відповідної фічі.

**Рекомендація:** не чіпати схему. Класифікацію лишити як бэклог на етап чистового
дизайну кожної фічі; переносити декларативно (валідації/процеси), доменні — вручну.
